# CapBench HW/AG Classification — Work Order

**Read `PROJECT_KNOWLEDGE.md` first** for the full context. This file is the actionable handoff for one specific task.

## Goal

For each of the 41 CapBench tasks listed below, classify every file in `base_repo/T<NN>/eval/gt_files.txt` as either **handwritten (HW)** or **auto-generated (AG)**, then write the two lists into:

- `base_repo/T<NN>/eval/handwritten_files.txt`
- `base_repo/T<NN>/eval/auto_generated_files.txt`

The two files together must cover every entry in `gt_files.txt` (no overlap, no omissions).

## Why this matters

The HW/AG split is the **denominator** for the paper's headline metric (HW file coverage rate). Currently 41/50 CapBench tasks have an empty `auto_generated_files.txt` and `handwritten_files.txt == gt_files.txt`, which inflates the apparent task size and makes some agents look worse than they are. Without a correct split we can't include CapBench in the cross-benchmark comparison Table 3 of the paper.

The 9 already-classified CapBench tasks (T12, T17, T31, T34, T38, T40, T43, T49, T50) and the 12 legacy tasks (C1-C5, M1-M3, K1-K4) are the reference. Their splits are considered correct.

## Methodology — independent dual-pass + consensus

For each unclassified task, run **two independent passes** and only commit when they agree:

1. **Pass A — Claude agent** (general-purpose subagent in this Claude Code session). Spawn one subagent per task with the prompt below.
2. **Pass B — Codex CLI** (`codex` invoked via the `codex:codex-rescue` agent or directly through `Bash`). Same prompt, no awareness of Pass A's answer.

Then run a **consensus step**:
- If A and B agree on every file → commit.
- If they disagree, run a **third arbiter pass** (another Claude subagent) showing both classifications and the file diffs. The arbiter writes the final list.

Why two independent agents: this is the same defense-in-depth strategy used for the K3 evaluator cross-validation. A single LLM hallucinates classifications confidently; two independent passes catch about 90% of those.

## Heuristics each agent should apply

Tell each agent these rules. They are non-trivial and tasks differ.

### Mark as **AUTO-GENERATED** if:
- Filename matches `go.sum`, `*.lock`, `package-lock.json`, `yarn.lock`, `Pipfile.lock`, `Gemfile.lock`, `Cargo.lock`, `composer.lock`
- Path contains `vendor/`, `node_modules/`, `third_party/`, `external/`
- File is a generated Go/Java/Python from `.proto`, `.thrift`, `.flatbuf`, etc. — usually `*.pb.go`, `*_pb2.py`, `*_grpc.pb.go`, `*_generated.go`, `zz_generated_*.go`
- File is OpenAPI/Swagger output: `swagger.json`, `openapi.yaml`, `api/openapi-spec/*`
- Path matches `**/generated/**`, `**/gen/**`, `zz_*`, `*.gen.*`
- File is a snapshot/golden file: `__snapshots__/`, `testdata/golden/`, `*.golden`, `*.snap`
- File is a build artifact: `dist/`, `build/`, `target/`, `out/`, `_output/`, `bazel-bin/`
- Header line matches `// Code generated by ... DO NOT EDIT.` or similar (Go convention)
- Python: file in `__pycache__/`, `*.pyc`, or `*_pb2.py`, `*_pb2_grpc.py`, `*_pyi.pyi` from protoc
- Comment in file says "auto-generated", "DO NOT EDIT", "generated by", etc.
- The PR diff for the file is mechanical (e.g., regenerated formatting, version bumps, hash updates)

To check the "DO NOT EDIT" header, the agent can use `git show <base_sha>:<file>` from inside the task's repo, or read the modified file from the diff hunk.

## Workflow per task

For each `<TASK>` in the unclassified set:

1. Read `base_repo/<TASK>/eval/gt_files.txt` (list of changed files, one per line).
2. For each file, decide HW vs AG using the heuristics above. When the heuristic isn't conclusive, open the file in `base_repo/<TASK>/repo/<file>` and look at the header / first 30 lines for "auto-generated", "do not edit", protoc/swagger markers, or similar.
3. For ambiguous files, also check the corresponding hunk in `gt_diff.patch` — if the hunk is purely whitespace/regen of a generated artifact, it's AG.
4. Write the two split files:
   - `base_repo/<TASK>/eval/handwritten_files.txt`
   - `base_repo/<TASK>/eval/auto_generated_files.txt`
5. Verify: `wc -l` of both files should sum to the line count of `gt_files.txt`.

## Two-agent consensus protocol

For each task, spawn **two independent agents** (one Claude general-purpose, one Codex CLI). Give them the **same prompt** and ask them to write their answers to scratch paths:

```
/tmp/capbench_classify/<TASK>/claude_hw.txt
/tmp/capbench_classify/<TASK>/claude_ag.txt
/tmp/capbench_classify/<TASK>/codex_hw.txt
/tmp/capbench_classify/<TASK>/codex_ag.txt
```

Then `diff` the two HW lists and the two AG lists.

- **If identical** → write to `base_repo/<TASK>/eval/{handwritten_files.txt,auto_generated_files.txt}`, mark task ✅ done.
- **If they disagree** → list the disputed files, present them to a third arbiter (Claude with full context including the file content of disputed files), and have the arbiter make the final call. Write the result.

A `disputed.json` per task with the list of disagreements should be saved alongside, so we can audit later.

## Per-task agent prompt template

Use this verbatim for both Claude and Codex when classifying one task:

```
You are classifying files from a software project as either HANDWRITTEN (human-authored source code, tests, configs, docs) or AUTO_GENERATED (machine-produced files: lockfiles, vendored deps, generated stubs, snapshots, OpenAPI/protobuf output, build artifacts, BUILD files in Bazel, .pb.go, *_generated.go, swagger.json, etc.).

The project: {SHORT_DESCRIPTION_FROM_CAPBENCH_CSV}
Repository root: base_repo/{TASK}/repo
Task PR diff: base_repo/{TASK}/eval/gt_diff.patch (you may inspect this for context)

Here is the full list of files touched by the PR (one per line):
{GT_FILES_LIST}

For EACH path, decide HW or AG. Use these rules in order:
1. If the path matches obvious AG patterns (lockfiles, *.pb.go, *_pb2.py, *.swagger.json, vendor/, *_generated*, *.snap, *.golden, BUILD.bazel) → AG.
2. If the file at base_repo/<task>/repo/<path> exists, peek at the first 5 lines: presence of "DO NOT EDIT", "Code generated by", "AUTOGENERATED", "@generated", "automatically generated" → AG.
3. If the diff for the file in gt_diff.patch is purely additive boilerplate (e.g. all hunks are 100+ identical lines of struct definitions matching pb-style code) → AG.
4. Otherwise → HW.

Output two newline-separated path lists: one for HW, one for AG. They must partition gt_files.txt exactly (no overlap, no omissions).

## Sanity check after each task

```bash
TASK=T01
HW=$(wc -l < base_repo/$TASK/eval/handwritten_files.txt)
AG=$(wc -l < base_repo/$TASK/eval/auto_generated_files.txt)
GT=$(wc -l < base_repo/$TASK/eval/gt_files.txt)
[ $((HW + AG)) -eq $GT ] && echo "$TASK OK ($HW HW + $AG AG = $GT)" || echo "$TASK MISMATCH"
```

## Heuristics that already worked

These are the patterns the 9 already-classified tasks taught us. Use them as priors:

- **Generated** if the path matches: `*.pb.go`, `*.pb.cc`, `*.pb.h`, `*_pb2.py`, `*_pb2_grpc.py`, `*.gen.go`, `zz_generated_*.go`, `bindata.go`, `*.swagger.json`, `*openapi*.json`, `swagger.json`, `*.lock`, `package-lock.json`, `yarn.lock`, `Cargo.lock`, `Gemfile.lock`, `*_generated.go`, `*.generated.ts`, `vendor/**`, `third_party/**`, `node_modules/**`, `**/zz_generated*`, `*.snap`, `__snapshots__/**`, `**/testdata/golden/**`
- **Generated** if first 5 lines of file contain `DO NOT EDIT`, `Code generated`, `AUTO-GENERATED`, `@generated`, `Generated by`
- **Handwritten** otherwise (source code, tests, prose docs)

For ambiguous cases (e.g. JSON fixtures), prefer **handwritten** unless there's an explicit generator script alongside.

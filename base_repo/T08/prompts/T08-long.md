# T08: JDK — Reimplement Core Reflection on Method Handles (JEP 416)

## Requirement — self-contained spec

Upstream specification (do not fetch): JEP 416, "Reimplement Core Reflection
with Method Handles" (https://openjdk.org/jeps/416). The text below
paraphrases what the JEP requires; treat it as the contract.

## Goal

Reimplement `java.lang.reflect.Method`, `Constructor`, and `Field` on top of
`java.lang.invoke.MethodHandle`, removing the existing implementation that
generates bytecode at run time. The new implementation must preserve the
observable semantics of the existing public reflection API — this is *not*
a change to `java.lang.reflect`, only to its underlying mechanism.

## Background, in JEP terms

Today, core reflection has two internal mechanisms for invoking a
reflective target:

- A **VM native** path used for the first few invocations (low warm-up cost).
- A **dynamically generated bytecode stub** ("`MagicAccessorImpl`" subclass)
  used after a threshold, to amortize JIT cost.

Field access has a similar split. `MagicAccessorImpl` is special: the VM
exempts it from access checks, verifier rules, and class-load checks so
that the generated stub can punch through them.

This duplicates work already done by `java.lang.invoke`, which has a
direct method-handle invocation path with comparable performance and
no special VM treatment. The goal of JEP 416 is to delete the dynamic
bytecode path and route reflective invocation through method handles
instead, while preserving the early-invocation native path until method
handles are available.

## Goals

- Remove the dynamic bytecode-stub path used by `Method.invoke`,
  `Constructor.newInstance`, `Field.get*`, and `Field.set*`.
- Replace it with a method-handle-based path that achieves comparable
  performance after warm-up.
- Keep the public behavior of `java.lang.reflect` unchanged: same
  exceptions, same access checks, same nullability and primitive
  unboxing/boxing rules.
- Continue to use the VM native path for early bootstrap, before
  `java.lang.invoke` is initialized.
- Preserve a fallback so the old implementation can be reactivated
  for diagnosis.

## Non-goals

- This is not a redesign of the `java.lang.reflect` API. The public API
  (signatures, exception types, behaviour for invalid arguments, etc.)
  must be unchanged.
- The JEP does not commit to any particular layout of internal classes.

## Behaviour to preserve

For every `Method`, `Constructor`, and `Field`, the new implementation
must preserve:

- Argument-count and type checks, including primitive widening and the
  same exception types as before (`IllegalArgumentException`,
  `NullPointerException`, `IllegalAccessException`,
  `InvocationTargetException`).
- Access checks honour the access flags of the reflected member and the
  module-graph rules already enforced by core reflection.
- Caller sensitivity: where the reflective member is itself
  caller-sensitive, the caller class as seen from the original
  reflective entry point is what is forwarded to the underlying member.
- Setting `setAccessible(true)` continues to bypass language-level access
  checks subject to the module access checks.

## Bootstrap concerns

`java.lang.reflect` is used very early in VM startup, before
`java.lang.invoke` is fully usable. The implementation must therefore be
able to fall back to the native path during early bootstrap and switch
to the method-handle path once `java.lang.invoke` is available.

## Compatibility / fallback

A system property is provided that allows users to revert to the
previous (bytecode-generating) implementation in case of regression:

```
-Djdk.reflect.useDirectMethodHandle=false
```

The default value uses the new method-handle-based path. Setting it to
`false` selects the legacy implementation. (The property is documented
as transitional; the JEP states the legacy implementation will be
removed in a future release.)

## Performance

The change must not regress significantly versus the bytecode-spinning
implementation in steady state, with the understanding that:

- Method-handle invocation can use less metadata and produces no extra
  generated classes, so footprint and startup are improved.
- Some deeply tuned microbenchmarks may show small regressions in tight
  loops where the legacy spinner had been heavily JIT-specialized.

## Acceptance

- All existing reflection tests in the JDK pass unchanged.
- The reflective dispatch path no longer generates per-target classes
  derived from `MagicAccessorImpl` for `Method`, `Constructor`, or
  `Field` invocations on the new path.
- The system property `jdk.reflect.useDirectMethodHandle=false` reverts
  to the old behaviour, and the legacy code paths still pass tests.
- Behaviour for boxing/unboxing, widening, narrowing, exception
  wrapping (`InvocationTargetException`), access checks, and varargs
  matches the legacy implementation.

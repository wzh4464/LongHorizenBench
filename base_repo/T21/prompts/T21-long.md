# T21: CPython — PEP 617: New PEG Parser

## Requirement (inlined from PEP 617)

*Upstream: https://peps.python.org/pep-0617/* — "New PEG parser for CPython". Everything required to complete the task is captured below; do not fetch the URL.

## 1. Summary

CPython historically uses a hand-written LL(1) parser generated by an in-tree LL(1) parser-generator. PEP 617 replaces it with a parser based on a Parsing Expression Grammar (PEG). The new grammar is more expressive: it supports unlimited lookahead, ordered alternatives, and grammar-level actions that build the AST directly without a separate concrete syntax tree. The end result is the same Python language with the same accepted programs, but produced by a different parser whose grammar is easier to evolve.

## 2. Why PEG

The LL(1) parser had several limitations:

* It could not handle ambiguous-looking constructs without grammar contortions; new syntax (assignment expressions, parenthesised `with`, soft keywords) required workaround productions.
* It produced a concrete syntax tree first and then converted it to AST in a second pass, which is slower and harder to maintain.
* Error messages were generated mechanically from the parse stack and were difficult to improve.

A PEG (Parsing Expression Grammar) parser:

* Allows ordered-choice alternatives, eliminating ambiguity by construction.
* Supports unlimited lookahead via packrat memoisation, which simplifies grammar rules.
* Can carry semantic actions inline so AST construction happens during parse.
* Produces parsers that are amenable to better error reporting (synchronisation points, expected-token sets).

## 3. Goals

1. Replace the LL(1) parser with a PEG-based parser generated from a single grammar file.
2. Preserve the existing AST shape and the public C and Python parser APIs.
3. Match or improve the parser's runtime performance (target: within ~10% of the LL(1) parser on a representative corpus).
4. Improve syntax-error messages, particularly around closing brackets, missing colons, and invalid targets in assignments.
5. Allow Python code to evolve more easily by adding rules that the LL(1) parser could not express.

## 4. Acceptance criteria

* All existing CPython tests pass.
* `ast.dump(ast.parse(src))` is byte-identical for any well-formed source on the new and old parsers.
* `python -m timeit -s 'import ast; src=open("Lib/test/test_grammar.py").read()' 'ast.parse(src)'` runs at most 10% slower than on the LL(1) baseline.
* Error messages for a curated set of malformed programs improve in clarity (compare with snapshot fixture committed alongside the change).
* The legacy LL(1) parser stays available behind `-X oldparser` for one release cycle, then is removed.

## 5. Out of scope

- Adding new language syntax.
- Changing the AST node hierarchy.
- Performance work beyond what the new parser already enables (e.g. peephole, bytecode reorganisation).
- Tooling consumers of the parser (linters, formatters) — they should continue to work via `ast`.

## 6. Notes for the implementor

The PEP itself describes the meta-grammar and meta-parser; the goal here is to apply that design to CPython. The implementor should derive the location of the parser modules and where to wire the new entry point from the existing source layout — the spec does not prescribe specific file names.
